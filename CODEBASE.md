## System Overview

This codebase controls a six-wheeled rocker-bogie rover built for NASA's HERA (Human Exploration Research Analog) simulation. An astronaut on a laptop uses a browser-based control panel to send commands to the rover over either WiFi or LoRa radio. Commands can be typed manually, selected from presets, or generated by an on-board AI assistant. The rover also has a ZED 2i stereo camera that streams a live 3D point cloud map back to the browser as the rover moves.

**Two transport modes:**
- **WiFi** — full bidirectional communication; all command types available; rover returns results in real time.
- **LoRa** — long-range radio; one-way transmission only; no response returned; limited to `bash_command`, `edit_file`, and `basic_action`.

## File-by-File Reference

---

### `server/FlaskServer.py`

**Runs on:** Astronaut laptop  
**Purpose:** The central server that ties everything together. Serves the browser UI, proxies commands to the rover, streams AI responses, and relays the 3D map from the Jetson to the browser.

#### Key globals

| Variable | Type | Purpose |
|---|---|---|
| `FASTAPI_SERVER_URL` | str | WiFi URL of the Pi's FastAPI server (`/execute`) |
| `TRANSMISSION_MODE` | str | `"wifi"` or `"lora"` — default mode |
| `LORA_MESSAGE_PATH` | str | File path where the LoRa transmitter reads its payload |
| `OLLAMA_MODEL` | str | Local LLM model name (e.g. `qwen3:0.6b`) |
| `JETSON_WS_URL` | str | WebSocket URL of the Jetson mapper (default `ws://192.168.1.100:9001`) |
| `_sse_clients` | `list[queue.Queue]` | One queue per open `/map_stream` SSE connection |
| `_jetson_ws_connected` | bool | Whether the background WS thread is currently connected to the Jetson |
| `_jetson_ws_handle` | WebSocketApp | Live WebSocket handle for forwarding control messages |
| `_map_point_count` | int | Cumulative points received since last clear |
| `_map_seq` | int | Last chunk sequence number received |

#### Endpoints

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/` | Serves `server/templates/index.html` |
| `POST` | `/send_command` | Forwards a rover command via WiFi or writes it to file for LoRa |
| `GET` | `/config` | Returns current server configuration as JSON |
| `POST` | `/config` | Updates server configuration (rover IP, timeout, mode, LoRa destination, Jetson WS URL) |
| `POST` | `/ai_command` | Streams an LLM-generated command via SSE (thinking tokens + final JSON) |
| `GET` | `/map_stream` | SSE stream of 3D map chunks from the Jetson → browser |
| `POST` | `/map_control` | Forwards `start`/`stop`/`clear` to the Jetson; resets counters on `clear` |
| `GET` | `/map_status` | Returns `{connected, point_count, seq}` |

#### Background thread: `_jetson_ws_thread()`

Runs at startup (unless `TESTING=1` env var is set). Maintains a persistent WebSocket connection to the Jetson mapper. On each incoming message it calls `_broadcast_to_sse()` to push the JSON to all open browser SSE connections. Auto-reconnects every 5 seconds on disconnect.

#### AI streaming (`/ai_command`)

Uses the `ollama` Python library with `think=True` and `stream=True`. Streams SSE events of types: `thinking_start`, `thinking`, `thinking_end`, `content`, `result`, `error`, `done`. The LLM is constrained to output a JSON object matching `RoverCommand` (type + fields). The system prompt is loaded from `ai_system_prompt.md` and the Rover API reference is appended at startup.

### `server/templates/index.html`

**Runs in:** Browser on astronaut laptop  
**Purpose:** The entire mission control UI — a single-page app with two tabs: Commands and Map.

#### Design system

Uses the METSAnauts website design language: `#070c16` background with a 28px dot-grid, `#5b9cf6` accent blue, Space Grotesk body font, JetBrains Mono for code/labels, `#0d1425` surface cards.

#### Tab: Commands

The original rover control panel:

- **Mode switcher** — WiFi / LoRa toggle. Switches which config fields are visible.
- **Config section** — Rover IP, timeout, LoRa destination, Jetson WS URL. `POST /config` on save.
- **Command type dropdown** — `bash_command`, `edit_file`, `basic_action`, `read_file`, `read_image`. Shows/hides the relevant input fields dynamically.
- **Autofill suggestions** — Quick-insert buttons for common commands (forward, turn, stop, etc.) that fill the action textarea.
- **Send button** — `POST /send_command`. Displays structured response fields (status, stdout, output, image preview, etc.).
- **AI Assist FAB** — Floating button opens the AI chat modal. Messages stream via `EventSource('/ai_command')`. When the LLM produces a `result` event, the command fields are auto-filled and the modal closes.

#### Tab: Map

Live 3D terrain mapping:

- **Status dot** — Green (CONNECTED) / Red (DISCONNECTED). Polled via `GET /map_status` every 3 seconds.
- **Start / Stop / Clear buttons** — Call `POST /map_control` and manage the SSE connection and elapsed timer.
- **Three.js canvas** (`#mapCanvas`) — 520px tall. Renders a growing point cloud using `THREE.Points` with pre-allocated `Float32Array` buffers (up to 600,000 points). Points are appended incrementally; the geometry's `drawRange` advances rather than rebuilding the full cloud. OrbitControls for rotate/pan/zoom.
- **Minimap** (`#minimapCanvas`) — 160×160px canvas overlay. Draws the rover's XZ path as a blue line with a white dot at current position. Scales dynamically to fit the path bounds.
- **Stats strip** — Point count, coverage area (m²), elapsed time (MM:SS).

#### Key JavaScript functions

| Function | What it does |
|---|---|
| `switchTab(tab)` | Shows/hides `#tab-commands` / `#tab-map`; calls `initThreeJS()` on first map switch |
| `initThreeJS()` | One-time setup of Three.js scene, camera, renderer, OrbitControls, BufferGeometry |
| `appendPoints(points)` | Writes `[x,y,z,r,g,b]` arrays into the pre-allocated Float32Array buffers, advances draw range, updates stats |
| `resetPointCloud()` | Zeros the draw range and stats |
| `openMapSSE()` | Opens `EventSource('/map_stream')`; on `chunk` events calls `appendPoints` + `updateMinimap` |
| `closeMapSSE()` | Closes the SSE connection |
| `updateMinimap(roverPos)` | Appends `[x,z]` to `roverPath`, redraws minimap canvas |
| `startMapping()` | Opens SSE, sends `start` to Flask, starts elapsed timer |
| `stopMapping()` | Sends `stop` to Flask, clears timer |
| `clearMapping()` | Sends `clear`, resets point cloud + minimap + stats |
| `_pollMapStatus()` | Fetches `/map_status`, updates status dot; runs every 3s + immediately on load |

### `rover_pi/main.py`

**Runs on:** Raspberry Pi  
**Purpose:** FastAPI server. The primary command execution endpoint when using WiFi mode.

Exposes one endpoint: `POST /execute`. Accepts a `CommandPayload` (type + optional fields) and delegates to `command_executor.execute_command()`. Also has `GET /` as a health check.

Run with: `uvicorn main:app --host 0.0.0.0 --port 8000` (from `rover_pi/`)

---

### `rover_pi/command_executor.py`

**Runs on:** Raspberry Pi (imported by `rover_pi/main.py` and `rover_pi/serial_reader.py`)  
**Purpose:** The single execution engine for all five command types. Both the HTTP server path and the LoRa serial path use this module so behavior is identical regardless of transport.

#### Command types

| Type | Function | What it does |
|---|---|---|
| `bash_command` | `execute_bash_command(command)` | Runs a shell command via `subprocess.run`, returns stdout/stderr/return code |
| `edit_file` | `execute_edit_file(file_name, file_content)` | Writes content to a file |
| `basic_action` | `execute_basic_action(action)` | Executes Python code via `exec()`, captures stdout |
| `read_file` | `execute_read_file(file_name)` | Reads and returns file contents as a string |
| `read_image` | `execute_read_image(file_name)` | Reads an image and returns it as base64 with MIME type |

All functions return a standardized dict: `{"status": "success"|"error", "type": ..., ...}`.

`execute_command(payload)` is the router — it reads the `type` field and dispatches to the right function.

### `hardware/Robot.py`

**Runs on:** Raspberry Pi  
**Purpose:** Hardware abstraction layer. Wraps GPIO motor control, servo control, and camera capture into clean Python classes. The Pi's command executor runs scripts that import this module. This is the original monolithic driver; `hardware/robot_base.py` + `hardware/robot_rpi.py` provide a newer split (abstract base + concrete impl) covering the same surface.

#### Classes

**`Drivebase`**  
Controls 4 DC motors (2 per side) via GPIO PWM at 1 kHz. Pins: IN1=22, IN2=24, IN3=17, IN4=27.
- `setLeft(power)` / `setRight(power)` — power: −1.0 (reverse) to 1.0 (forward)
- `drive_instant(left, right)` — set both motors continuously
- `drive(left, right, duration)` — blocking drive then stop
- `forward(power, duration=None)`, `turn_left(...)`, `turn_right(...)`, `stop()` — convenience methods
- `cleanup()` — stops PWM and releases GPIO

**`RockerBogie`**  
Controls 4 servos via the Pi Servo Hat (I²C).
- `setPositions([a, b, c, d])` — set all 4 servos in degrees
- `toRegularPosition()` — `[90, 90, 150, 30]` — standard driving pose
- `toSunPosition()` — `[150, 35, 90, 90]` — tilted for solar panel orientation

**`Camera`**  
Wraps OpenCV `VideoCapture`. Lazily imported (`cv2` may not be installed).
- `capture()` — returns a numpy BGR frame
- `save_as_png(image, filepath)` — writes `{filepath}.png`
- `capture_and_save(filepath)` — capture + save, returns path
- `release()` — frees the camera

**`Rover`**  
Composite class combining `Drivebase`, `RockerBogie`, and `Camera`. This is the class AI-generated `basic_action` scripts instantiate.
- All `Drivebase` methods are delegated through (e.g. `rover.forward(0.7, duration=2)`)
- All `RockerBogie` methods are delegated through (e.g. `rover.setup_regular_position()`)
- `init_camera(index=0)` — initialize camera on demand
- `take_picture(filepath)` — auto-inits camera if needed, captures and saves
- `cleanup()` — must be called at end of every `basic_action` script to release GPIO

### `hardware/robot_base.py` / `robot_rpi.py` / `robot_jetson.py`

**Runs on:** Raspberry Pi (`robot_rpi.py`) and Jetson Nano (`robot_jetson.py`)  
**Purpose:** A split of the monolithic `Robot.py` into an abstract base (`robot_base.py`) plus per-platform concrete implementations. `robot_rpi.py` provides the same Drivebase/RockerBogie/Camera/Rover surface as `Robot.py` on the Pi; `robot_jetson.py` provides the Jetson-side rover driver and houses the ZED spatial-mapping logic described below.

### `hardware/robot_jetson.py`

**Runs on:** Jetson Nano  
**Purpose:** Streams a growing 3D point cloud to the Flask server over WebSocket. Uses the ZED SDK's spatial mapping feature to build a fused point cloud as the rover drives, then sends only new voxels each cycle.

#### Module-level functions (pure numpy, no ZED required)

**`get_voxel_key(x, y, z, voxel_size=0.05)`**  
Returns an `(ix, iy, iz)` integer tuple identifying which 5cm voxel cell a point falls in. Uses `np.floor(coord / voxel_size)`.

**`voxel_downsample(vertices, colors, voxel_size=0.05)`**  
Reduces an Nx3 point cloud to one point per voxel cell. Returns `(filtered_vertices, filtered_colors, voxel_keys_set)`. Pure numpy — no Open3D dependency (important for Jetson ARM64 compatibility).

**`height_colors(vertices)`**  
Generates Nx3 float32 RGB colors using a blue→green→red gradient mapped to the Y (height) axis. Blue = lowest point, green = mid, red = highest. Used because extracting per-vertex RGB from the ZED mesh is complex; height coloring is also more useful for terrain visualization.

#### `VoxelTracker`

Maintains a Python `set` of all voxel keys already transmitted. `get_new_points(vertices, colors)` returns only points in new voxels (as `[x, y, z, r, g, b]` with r/g/b as 0–255 ints), and adds them to the sent set. `clear()` resets the sent set so the next call returns everything again.

#### `ZedMapper`

The main class. Manages the ZED camera, WebSocket server, and mapping loop.

**`_init_zed()`** — Opens the ZED with `ULTRA` depth mode, Y-up right-handed coordinates, enables positional tracking and spatial mapping at 5cm resolution. Raises `RuntimeError` on any failure.

**`_get_rover_pos()`** — Queries the ZED's positional tracking to get current `[x, y, z]` in world meters.

**`_extract_points()`** — Calls `extract_whole_spatial_map()` to get the full fused mesh, computes height colors, then voxel-downsamples.

**`_mapping_loop()`** — Async loop at ~2 Hz. On each tick (if `mapping_active`): grabs a ZED frame, extracts points, gets new points from `VoxelTracker`, broadcasts a `chunk` message if any exist.

**`_handle_client(websocket)`** — Handles one WebSocket client. Accepts `{"action": "start"|"stop"|"clear"}` messages. On `clear`: resets tracker, resets seq, calls `zed.clear_spatial_map_async()`.

**`_broadcast(message)`** — Sends to all connected clients using `asyncio.gather`. Removes stale clients that raise exceptions.

**`run()`** — Inits the ZED, starts the WebSocket server on `0.0.0.0:9001`, runs the mapping loop concurrently.

**WebSocket message format:**
```json
{
  "type": "chunk",
  "seq": 42,
  "rover_pos": [1.23, 0.05, -0.87],
  "points": [[x, y, z, r, g, b], ...]
}
```

### `communications/lora/transmitter_lora.py`

**Runs on:** LoRa transmitter module (CircuitPython, e.g. Adafruit Feather M0 RFM95)  
**Purpose:** Reads `message.json` from its local filesystem and transmits it over 915 MHz radio as a series of fragmented packets.

**Packet structure** (248-byte payload after library strips its 4-byte header):
- Byte 0: Recipient ID (1 byte)
- Bytes 1–4: Packet Content ID (unique 4-byte int per transmission)
- Bytes 5–6: Chunk index (2 bytes)
- Bytes 7–8: Total chunks (2 bytes)
- Bytes 9–247: JSON data chunk (239 bytes, zero-padded on last chunk)

`create_packets(json_data, recipient_id, packet_content_id)` slices the JSON bytes into 239-byte chunks and builds the packet bytearray for each.

The `message.json` file is written by `server/FlaskServer.py`'s `_send_lora()` function on the laptop, which a human physically copies to the LoRa module's drive (or the module watches a shared path).

### `communications/lora/receiver_lora.py`

**Runs on:** LoRa receiver module (CircuitPython, on the Pi or nearby)  
**Purpose:** Listens for RFM95 packets at 915 MHz, reassembles multi-chunk messages, and prints the complete JSON to its USB serial output prefixed with `DATA_JSON:`.

`parse_packet(packet)` — extracts header fields from a raw packet bytes object.

`reassemble_message(packet_content_id)` — once all chunks for a content ID are buffered, strips null padding and decodes as JSON.

The main loop:
- Receives packets with a 5-second timeout
- Filters by `MY_ID` (ignore packets for other recipients; accept broadcast `255`)
- Buffers chunks by `packet_content_id`
- When all chunks arrive: prints `DATA_JSON:{json}` on serial
- Cleans up incomplete messages older than 60 seconds

### `rover_pi/serial_reader.py`

**Runs on:** Raspberry Pi  
**Purpose:** Bridges the LoRa receiver to the rover's command executor. Watches the USB serial port of the LoRa receiver module, extracts `DATA_JSON:` lines, and executes the commands locally.

Only supports `bash_command`, `edit_file`, and `basic_action` (no `read_file`/`read_image` — LoRa is one-way, can't return data).

- `find_circuitpython_port()` — auto-detects the CircuitPython serial port by checking description/manufacturer strings.
- `handle_command(payload)` — validates the command type against the LoRa-allowed list, then delegates to `command_executor.execute_command()`.
- Main loop — reads lines from serial, finds `DATA_JSON:` prefix, parses JSON, executes command, logs result. Auto-reconnects if serial drops.

Usage: `python serial_reader.py [--port COM5] [--baud 115200] [--verbose]` (from `rover_pi/`)

---

### `rover_pi/rover_receiver.py`

**Runs on:** Raspberry Pi  
**Purpose:** UDP gamepad listener. Receives joystick/button packets from the laptop-side `communications/laptop_controller.py` and drives the rover hardware directly. Manipulates `sys.path` at import time so it can reach `hardware/` from `rover_pi/`.

---

### `communications/laptop_controller.py`

**Runs on:** Astronaut laptop  
**Purpose:** Gamepad → UDP bridge. Reads a connected gamepad on the laptop and forwards control packets to `rover_pi/rover_receiver.py` over UDP. Pairs with `rover_receiver.py` to provide a direct-drive control loop independent of the Flask command pipeline.

---

### `server/ai_system_prompt.md`

**Loaded by:** `server/FlaskServer.py` at startup  
**Purpose:** System prompt for the local LLM (Ollama). Instructs the model to translate plain-English rover requests into one of the five structured command types.

Key instructions:
- Prefer `basic_action` for any physical rover operation (movement, servos, camera) — not `bash_command`
- Every `basic_action` must start with `from Robot import *`, instantiate `Rover()`, and call `rover.cleanup()` at the end
- Output must be a JSON object: `{"type": ..., "fields": {...}}`
- In LoRa mode, never use `read_file` or `read_image`
- The Rover class API reference is appended dynamically by `server/FlaskServer.py`

## Data Formats

### Command payload (browser → Flask → Pi)

```json
{
  "type": "basic_action",
  "fields": {
    "action": "from Robot import *\nrover = Rover()\nrover.forward(0.5, duration=2)\nrover.cleanup()"
  },
  "mode": "wifi"
}
```

For LoRa mode, `FlaskServer` writes a `message.json` with a `Recipient` field:
```json
{
  "Recipient": 0,
  "type": "basic_action",
  "fields": { "action": "..." }
}
```

### Map chunk (Jetson → Flask → Browser via SSE)

```json
{
  "type": "chunk",
  "seq": 42,
  "rover_pos": [1.23, 0.05, -0.87],
  "points": [
    [x, y, z, r, g, b],
    ...
  ]
}
```

- `x, y, z` — meters, ZED world frame (right-handed, Y-up)
- `r, g, b` — 0–255 integers (height-mapped: blue=low, green=mid, red=high)
- Typical chunk: 500–2000 points at 5cm voxel spacing
- Keepalive (when idle): `{"type": "keepalive"}`

## Configuration & Deployment

### Astronaut laptop

```bash
pip install flask requests ollama pydantic websocket-client
python server/FlaskServer.py
# or: server/run_server.bat
# Open http://127.0.0.1:5000
```

Set the rover's IP, Jetson IP, and LoRa destination in the Config section of the UI. These are persisted in-memory and reset on server restart (no config file currently).

### Raspberry Pi (WiFi mode)

```bash
pip install fastapi uvicorn pydantic
cd rover_pi
uvicorn main:app --host 0.0.0.0 --port 8000
```

### Raspberry Pi (LoRa mode)

1. Copy `communications/lora/receiver_lora.py` as `code.py` onto the LoRa receiver module.
2. The module will print received JSON to USB serial prefixed with `DATA_JSON:`.
3. On the Pi, run: `python rover_pi/serial_reader.py`
4. On the laptop, set the mode to LoRa in the UI and set the LoRa message path to wherever the transmitter reads from.

### Jetson Nano (3D mapping)

```bash
pip install websockets numpy
# pyzed comes with the ZED SDK installer
python hardware/robot_jetson.py
```

Then set the Jetson WS URL in the Flask UI Config section (`ws://<jetson-ip>:9001`), switch to the Map tab, and click Start.

## Testing Files

The `testing_files/` directory contains standalone scripts used during hardware development:

| File | Purpose |
|---|---|
| `LDR.py` | Reads a light-dependent resistor via GPIO analog input |
| `PlotLDR.py` | Plots LDR readings over time using matplotlib |
| `MPU6050.py` | Driver for the MPU-6050 6-axis IMU over I²C |
| `MPU-Interpreter.py` | Reads MPU6050 and prints roll/pitch/yaw |
| `RockerBogieSubsystemTest.py` | Manually tests all 4 rocker-bogie servo positions |
| `ServoTest.py` | Sweeps servos through their full range for calibration |

These are not part of the deployed system and are not imported by any production code.
